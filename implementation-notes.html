Implementation status

All endpoints and behaviors required by R2 are shipped and exercised at multiple test layers.

  • GET /runs — cursor-paginated list with filters and a Zod-driven 400 invalid_query envelope.
  • GET /providers, GET /runtimes, GET /models — cursor pagination, provider-slug + adapterType filters where applicable.
  • GET /artifacts/:id + GET /artifacts/:id/content — global lookup and raw stream with per-type Content-Type.
  • GET /runs/:id/events — replay-only, bounded stopAfter, and open-ended live=1 with 15s heartbeat, 5min idle close, Last-Event-ID resumption, and 1s disconnect cleanup.
  • Unified error envelope retrofitted across all 4xx/5xx responses.
  • Route tests in packages/protocol-rest/test, SSE tests in packages/protocol-sse/test, real-HTTP smoke and leak test in apps/daemon/test.
  • pnpm typecheck, pnpm test, pnpm build, and pnpm lint all pass after the changes.

Design decisions

Slug-to-id resolution for registry filters

decision The registry has no separate slug column. For /runtimes?provider=openai and /models?provider=openai the slug openai is resolved by prepending provider_ if missing, matching the seed convention (provider_openai, runtime_codex, model_gpt_5_5). The same shortcut works for single-record lookups, so /providers/openai resolves to provider_openai.

For the /runs filter, slug values are passed through unchanged because the runs table stores user-supplied slug-like strings ("openai", "test") directly, not registry ids.

Opaque cursor encoding

decision Cursors are base64url-encoded JSON of the keyset tuple ({createdAt, id} for runs, {id} for registry lists). Encoding/decoding lives in packages/contracts/src/cursor.ts so every endpoint shares one implementation. Malformed cursors return 400 invalid_query with details[0].path = "before".

Open-ended SSE writes raw to the Node response

decision reply.hijack() hands the underlying socket to streamRunEvents. The streamer holds an EventBus subscription, a 15s heartbeat interval, and a 5min idle setTimeout. Any of disconnect, idle, or stopAfter triggers a single close() that clears all three. The cleanup path is verified by a 50-iteration connect/abort leak test.

Zod issue path → details path

decision The spec example expects details[0].path = "status" for a bad CSV value. Zod produces ["status", 0] because it points at the failing array element. The serializer in packages/protocol-rest/src/http-errors.ts truncates the trailing numeric segment so the path always names the parameter the caller sent.

In-memory daemon still exposes /artifacts/:id

decision When the daemon runs without persistent storage (no config, used by some tests), metadata lookups still work, but GET /artifacts/:id/content returns 404 missing_artifact_content because there is no filesystem reader. The error path goes through the same envelope helper, so the response shape matches production.

SSE leak test uses FD count, not internal subscriber count

decision apps/daemon/test/sse-leak.test.ts connects 50 times via real HTTP, aborts each request, and asserts that the open file descriptor count does not grow by more than 10 above baseline (a conservative threshold to absorb logger/sqlite IO noise). The per-connection subscriber assertion (subscribers.size === 0) is verified separately in packages/protocol-sse/test/open-ended-stream.test.ts against a synthetic destination.

Deviations from the spec

Routes not in any closed code use internal_error

deviation The closed error-code table in the spec does not contain a code for "no such route", so the daemon's Fastify notFoundHandler returns 404 with the envelope { error: { code: "internal_error", message: "Route GET /foo not found" } }. This pairs 404 with internal_error, which the table maps to 500; the spec also says "Replaces any default Fastify error payload" for internal_error, which is the rule we follow here. Listed in open questions in case a dedicated route_not_found code is preferred.

R2 has no orphan-models exclusion case to actually exercise

deviation The spec says GET /models without a provider filter must include "orphan" models whose provider is not yet seeded. The current modelSchema requires a providerId, so the schema makes orphans impossible. The implementation still applies the correct filter semantics — when provider is omitted no provider filter is applied; when present, the filter is enforced — so the moment the schema relaxes (R3+), the behavior is already correct. Documented as an open question whether the schema should be relaxed in R2.

One R1 test was retargeted

deviation packages/protocol-rest/test/run-routes.test.ts had a test "treats invalid stopAfter as replay-length fallback" that exercised the bounded-collector path with live=1&stopAfter=not-a-number. Under R2, live=1 without a usable stopAfter means the connection stays open until disconnect or idle close — Fastify inject() cannot model a TCP disconnect, so the request would block. The test was rewritten to assert the same invalid-stopAfter fallback against the replay-only mode (no live=1), which still terminates deterministically. Open-ended live behavior is exercised in the new SSE tests instead.

Tradeoffs considered

DecisionPickedAlternativeWhy
SSE transport Raw reply.hijack() + res.write/res.end. A custom Fastify SSE plugin (e.g. fastify-sse-v2). The daemon already had a hand-rolled formatter in protocol-sse; adding a plugin would couple us to its lifecycle (its own heartbeat, its own resumption story) and complicate the leak test. Raw response gives us complete control over heartbeat, idle, and disconnect.
Slug resolution Strip-prefix shortcut: openai → provider_openai. Add a dedicated slug column to the registry tables and resolve via lookup. Spec is R2 and explicitly out-of-scope for new registry columns. The prefix-strip is reversible and deterministic, and it leaves the registry schema untouched until R3.
Cursor payload JSON-encoded keyset tuple, base64url, no signature. HMAC-signed cursors or a server-side cursor store. Local daemon only, 127.0.0.1 bound, no auth/rate-limiting in scope; opaque cursors with a documented "reserve the right to change encoding" disclaimer match the spec's stated contract.
Error envelope retrofit Fastify setErrorHandler + explicit sendHttpError helper, plus a typed HttpProblem exception class. A per-route try/catch that wraps every handler manually. The middleware approach guarantees coverage of unhandled throws (any unexpected throw becomes 500 internal_error with the envelope) and centralises Zod issue formatting so route code stays clean.
Leak detection FD-count delta after 50 cycles + subscriber-count test against synthetic destination. A dedicated probe that counts EventBus subscribers through the running daemon. EventBus isn't exported from the daemon, and exposing it just for tests would break encapsulation. The two-layer approach gives strong coverage without leaking internals.
List-routes test harness A small in-test factory that wires the same in-memory stores the daemon uses. Reuse createDaemonApp with an in-memory config flag. The daemon factory also wires Codex probing, run reconciliation, and registry seeding — heavier than needed and could mask issues by always succeeding. The hand-rolled factory keeps each test focused.

Test coverage notes

  • packages/protocol-rest/test/list-routes.test.ts: pagination round-trip, status/runtime/since filters, empty envelopes, all 400 invalid_query paths (bad status, since > until, limit > max, malformed cursor), provider-slug → registry-id resolution.
  • packages/protocol-rest/test/artifact-routes.test.ts: 200 metadata, 404 not-found, 200 content with application/x-ndjson, 404 missing_artifact_content for both metadata.contentStored=false and missing backing file, content-type picker spot-check.
  • packages/protocol-rest/test/registry-routes.test.ts: legacy single-record routes still emit the envelope; provider_not_found, runtime_not_found, model_not_found all carry the new shape.
  • packages/protocol-rest/test/run-routes.test.ts: error envelope on 409 adapter_protocol_failed and other paths, replay-only stream, bounded live stream, replay truncation, fractional stopAfter handling, the rewritten invalid-stopAfter case.
  • packages/protocol-sse/test/open-ended-stream.test.ts: replay-only end, bounded close at N, replay-then-live, Last-Event-ID resumption, 15s heartbeat + 5min idle close on a manual clock, subscriber release on client close.
  • apps/daemon/test/smoke.test.ts: real HTTP smoke against every shipped local endpoint, asserts relative changes ("list grew by 1") so the suite tolerates any pre-existing state.
  • apps/daemon/test/sse-leak.test.ts: 50 connect/abort cycles, FD growth bounded, daemon still healthy.

Open questions

Should "no such route" get its own closed code instead of internal_error?

Today the Fastify notFoundHandler returns 404 internal_error. The spec's closed code set doesn't include route_not_found, and internal_error's description ("Replaces any default Fastify error payload") is the closest match. If a future caller wants to disambiguate a typo'd URL from a real internal failure, the easiest move is to add a route_not_found entry. Flagged for confirmation.

Should modelSchema relax providerId in R2 to make orphan models representable?

The spec describes orphan-models behavior for GET /models but the schema currently requires providerId. The filter behavior is already correct in case the schema relaxes; the only question is whether R2 should make the schema change now, or leave it for R3 with capability records.

Are slug shortcuts on single-record routes (e.g. /providers/openai) acceptable?

The spec only mentions slug filters on list endpoints, but the single-record routes accept either an id or its prefix-stripped slug because the same translator is used both places. This is convenient and consistent with the close-set message "Unknown provider id or slug". Flagging in case strict id-only behavior was intended.

SSE leak threshold (FDs growth ≤ 10) — is the budget right?

On macOS/Linux the test asserts FDs do not grow by more than 10 above baseline after 50 connect/abort cycles. The threshold absorbs better-sqlite3 + Fastify logger churn. If CI runs on a smaller platform or the budget proves flaky, we may need to tighten or relax it.

Should Last-Event-ID override events that are still in the live stream?

Today, Last-Event-ID only trims the replay payload. If a client reconnects with a stale id, replay restarts from after that id and live events flow as usual. This matches typical EventSource semantics but isn't called out in the spec. Flagging for explicit confirmation.
